# Phase 1 — Data Model: Cloud Sync Backend

Authoritative field lists for every entity after applying the Phase 0 schema delta. Field names are camelCase (Prisma convention, server CLAUDE §6); Postgres column names are generated by Prisma without explicit `@map`.

## Enums

```prisma
enum Role {
  COLLECTOR
  COORDINATOR
}

enum CollectionStatus {
  SYNCED
  REJECTED
}
```

`CollectionStatus` reduced from the three-value scaffold to two values per FR-025. Migration renames the enum and drops `COMPLETED` (safe — no production data yet; constitution §IV conditionally passes).

## LessonInstance

| Field | Type | Nullable | Default | Notes |
|---|---|---|---|---|
| `id` | `String @id` | No | `uuid()` | — |
| `date` | `DateTime @db.Date` | No | — | Calendar date, no time component. |
| `seriesCode` | `String` | No | — | Copied from submission (soft ref — a collection may arrive before the Series row exists; the Series upsert happens in the same tx). |
| `topicId` | `String?` | Yes | — | FK → `LessonTopic.id`. |
| `professorId` | `String?` | Yes | — | FK → `Professor.id`. |
| `aggStart` | `Float?` | Yes | — | Median of normalized `attendanceStart` over eligible collections. |
| `aggMid` | `Float?` | Yes | — | — |
| `aggEnd` | `Float?` | Yes | — | — |
| `aggDist` | `Float?` | Yes | — | Median of `uniqueParticipants` (no normalization). |
| `aggCollectorCount` | `Int` | No | `0` | Number of eligible collections that fed the aggregate. |

**Unique**: `@@unique([date, seriesCode, topicId])` — identity derives from the triple, matching spec §Key Entities.

**Additional partial unique index (raw SQL in migration)**: Postgres treats NULL as distinct in a standard unique index, so the Prisma `@@unique` above does NOT prevent duplicate rows where `topicId IS NULL`. To make `(date, seriesCode)` truly unique for topicless instances, add:

```sql
CREATE UNIQUE INDEX lesson_instance_date_series_no_topic_key
  ON "LessonInstance"("date", "seriesCode")
  WHERE "topicId" IS NULL;
```

Without this index, two sync batches that each fail to resolve a topic would auto-create separate `LessonInstance` rows on the same day/series, splitting the aggregate.

**Index**: `@@index([date])` for date-range queries (FR-040).

**Invariants**:
- If `aggCollectorCount = 0`, every `agg*` field MUST be `null` (FR-025 "empty" rule via research §17). Enforced in `aggregateService.recompute`.
- Serializable per-instance via `pg_advisory_xact_lock(hashtext(id))` inside recompute transactions (research §9).

```sql
ALTER TABLE "LessonCollection"
  ADD CONSTRAINT rejection_reason_required
  CHECK (("status" = 'REJECTED' AND "rejectionReason" IS NOT NULL)
      OR ("status" = 'SYNCED'   AND "rejectionReason" IS NULL));
```

## State & Transitions

- **User.accepted**: `true ↔ false` via `PATCH /users/:id/accepted`. Each transition triggers `moderationService.cascadeRecompute(userId)` inside the same transaction (FR-051, US-5 scenario 3).
- **Collection.status**: `SYNCED` and `REJECTED` are both terminal from the client's perspective — no MVP transition. Server decides on ingest; later re-sends of the same id with newer `clientUpdatedAt` can flip a `REJECTED` collection back to `SYNCED` if the client corrected the malformed data, since FR-021 says "re-posting the same id applies field-level merge; newer wins".
- **Catalog.isPending**: `true → false` via any coordinator mutation (`POST`/`PATCH`). Sync upserts MUST NOT set `isPending = true` on an existing curated row (research §10).

## Referential integrity rules

- `LessonCollection.lessonInstanceId`: `ON DELETE RESTRICT`. Coordinators cannot delete instances in MVP (out of scope); if added later, the service must first `DELETE FROM LessonCollection WHERE lessonInstanceId = $1`.
- `LessonCollection.collectorUserId`: `ON DELETE RESTRICT`. Prevents accidental loss of history when a coordinator deletes a user (also out of scope for MVP).
- `LessonInstance.topicId`, `LessonInstance.professorId`: nullable FK with `ON DELETE RESTRICT` to back FR-034 (delete refused with conflict when referenced).

## Migration plan (first Prisma migration)

One migration named `0001_sync_backend_schema` produces the final shape. Because `prisma/migrations/` is empty today and no data exists in production, this is a rewrite rather than an in-place alteration; it still uses additive DDL so the same SQL is safe to re-run on an empty DB.

Migration contents:
1. `CREATE TYPE "Role"`.
2. `CREATE TYPE "CollectionStatus"` with two values `SYNCED`, `REJECTED`.
3. `CREATE TABLE` for User, LessonSeries, LessonTopic, Professor, LessonInstance, LessonCollection with the fields above.
4. `CREATE UNIQUE INDEX` for User.email, LessonSeries.code, Professor.email (partial, WHERE email IS NOT NULL), LessonInstance.(date, seriesCode, topicId).
5. `CREATE UNIQUE INDEX lesson_instance_date_series_no_topic_key ON "LessonInstance"(date, "seriesCode") WHERE "topicId" IS NULL` (closes the NULL-is-distinct hole described above).
6. `CREATE INDEX` for LessonTopic.(seriesId, sequenceOrder), LessonInstance.date, LessonCollection.lessonInstanceId, LessonCollection.collectorUserId.
7. `ALTER TABLE "LessonCollection" ADD CONSTRAINT rejection_reason_required CHECK (...)` (see above).
